Use this guide when you need to keep item data current and user behavior signals flowing into NeuronSearchLab.

## Setup

1. Define canonical prefixed item identifiers (`id`, for example `itm_7f3a2c9e`) in your source catalog.
2. Map source fields to `/v1/items` payloads (`name`, `description`, `metadata`).
3. Emit user events to `/v1/events` from all recommendation surfaces.

## Common workflows

### Create catalog items

```bash
curl -X POST https://api.neuronsearchlab.com/v1/items \
  -H "Authorization: Bearer <access_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "itm_7f3a2c9e",
    "name": "Nike Running Shoes",
    "description": "High-performance running shoes.",
    "metadata": { "category": "Footwear", "brand": "Nike", "price": 109.99 }
  }'
```

### Submit user interaction events

```bash
curl -X POST https://api.neuronsearchlab.com/v1/events \
  -H "Authorization: Bearer <access_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "user_id": "usr_456",
    "type": "click",
    "item_id": "itm_7f3a2c9e",
    "context_id": "ctx_101",
    "occurred_at": 1777478400,
    "click": {}
  }'
```

### Maintain ingestion quality

- Validate required fields before publishing payloads.
- Use Unix timestamps in integer seconds for event consistency.
- Monitor event throughput and lag in your own pipeline metrics.

## Example app journey: Daily sync + real-time feedback

1. Nightly job creates new catalog items using `/v1/items` and updates existing records with `/v1/items/{id}`.
2. App session requests recommendations from `/v1/recommendations`.
3. Frontend interaction events are forwarded to backend.
4. Backend publishes click/purchase events to `/v1/events` in near real time.
5. Team reviews Analytics to confirm engagement trends are improving.
